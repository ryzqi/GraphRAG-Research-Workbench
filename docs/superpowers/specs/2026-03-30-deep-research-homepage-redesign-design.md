# 深度研究首页重设计设计说明

- 日期：2026-03-30
- 主题：深度研究首页与前置规划流程重设计
- 状态：已确认设计，待进入 implementation plan

## 1. 背景

当前深度研究页已开始向 workbench 结构演进，但仍存在两个核心问题：

1. 首页入口仍偏“配置表单 + 立即启动”，不符合深度研究场景下“先澄清、再计划、再执行”的真实用户心智。
2. “执行前确认”仍是一个可选开关，而不是唯一主路径，导致产品行为不稳定，用户也难以理解系统到底何时会直接执行、何时会停下来等待确认。

本次重设计要把首页、澄清流程、计划确认和执行工作台收敛为一条清晰且唯一的交互路径，并把视觉风格从“功能型后台页面”提升为更接近 ChatGPT / Gemini / Qwen 深度研究入口的成熟研究体验。

## 2. 目标

### 2.1 产品目标

- 把深度研究入口改造成“研究前规划界面”，而不是“直接发起执行的表单页”。
- 固化主流程：**先澄清问题 -> 再生成计划 -> 用户确认后才执行研究**。
- 让首页在空态时简洁、安静、有研究感；在执行后再切换到信息密度更高的工作台。

### 2.2 体验目标

- 初次进入页面时，用户只看到最必要的输入入口，不被复杂配置压住。
- 当问题不清晰时，系统像一个真正的研究助手一样主动追问，而不是直接用模糊输入开跑。
- 计划确认阶段要有足够的“可读性”和“可判断性”，用户能看懂将要做什么，再决定是否执行。
- 确认后切入执行工作台，形成明确的阶段切换，增强“研究开始了”的感知。

### 2.3 工程目标

- 前端不再暴露“是否执行前确认”的可选配置。
- 前后端契约改为以“澄清 / 计划确认”为主线，避免双轨逻辑。
- 不保留“直接跳过计划确认”的兼容路径，除非未来产品重新明确要求。

## 3. 非目标

- 不在本次设计中扩展新的检索源、排序策略或 research runtime 能力。
- 不重做执行态全部细节，只在现有 workbench 基础上调整其接入时机和视觉承接关系。
- 不把深度研究页做成营销型 landing page。
- 不引入“用户可切换自动执行 / 手动确认”的模式开关。

## 4. 唯一事实源与核心约束

### 4.1 唯一事实源

本次设计的唯一事实源是以下已确认用户要求：

1. **始终先生成计划。**
2. **在提问不清晰的情况下，先反问用户并收集更多信息。**
3. **澄清追问发生在生成计划之前。**
4. **计划生成后必须等待用户确认执行。**
5. **首页空态采用极简方案。**
6. **视觉方向采用暗色、安静、编辑感。**
7. **计划以助手消息形式出现在对话流中。**
8. **确认后切换到研究执行工作台。**

### 4.2 设计约束

- 视觉参考可借鉴 ChatGPT / Gemini / Qwen 深度研究产品，但不能照搬品牌样式。
- 页面必须保持可读性和克制感，避免夸张科技风、重面板感和无意义动效。
- 所有关键决策都必须能映射到现有前端组件和后端会话状态，不做无法落地的设计。

## 5. 目标用户与使用场景

### 5.1 目标用户

- 需要做方案比较、技术调研、采购判断、路线分析的专业用户。
- 他们往往问题本身并不完全清晰，需要系统先帮助缩小研究边界。

### 5.2 典型场景

- “帮我对比几个 AI 编程工具并给团队采购建议。”
- “研究某项技术路线是否适合在未来 12 个月内落地。”
- “围绕某个复杂主题生成带证据的研究结论。”

这些场景的共同点是：用户的第一句话通常不够精确，直接执行常常会浪费额度、偏离重点，甚至得出不可用结论。

## 6. 总体交互模型

页面分为两个大阶段：

1. **确认前阶段：对话式研究规划**
2. **确认后阶段：执行工作台**

其中确认前阶段再细分为三个状态：

- 空态入口
- 澄清对话
- 计划确认

### 6.1 主流程

```text
进入深度研究页
-> 用户输入研究问题
-> 系统判断信息是否充分
   -> 若不充分：进入澄清对话，继续追问
   -> 若充分：生成研究计划
-> 展示计划并等待用户确认
-> 用户确认
-> 切换到执行工作台
-> 开始研究执行、展示进度与结果
```

### 6.2 关键原则

- 未确认前，页面永远停留在“规划阶段”。
- 只有确认后，才允许 runtime 真正启动。
- 澄清和计划确认都属于“研究前动作”，不应被包装成已经开始执行。

## 7. 页面信息架构

## 7.1 阶段一：空态入口

### 布局内容

- 页面标题
- 一句简短说明
- 主输入框
- 少量示例问题

### 设计特征

- 极简、单列、中心化布局
- 大标题突出“研究规划”而非“聊天”
- 输入框足够大，支持复杂问题直接粘贴
- 示例问题只作为帮助启发，不形成卡片墙或大段功能介绍

### 不应出现的内容

- 知识库勾选列表
- 允许外部研究之类的低层配置
- “执行前确认计划”开关
- 大量产品说明、卖点区块、统计数字或花哨图形

## 7.2 阶段二：澄清对话

### 触发条件

- 用户问题缺少关键范围、对象、维度、时间边界、输出目标之一或多个。

### 展现方式

- 页面原地转为对话流
- 用户原始问题显示为用户消息
- 系统以助手消息形式提出澄清问题
- 用户继续回答，直至信息充分

### 体验目标

- 让用户感到自己是在和研究助手共同“定义任务”
- 追问数量控制在必要范围，优先问最影响计划质量的问题
- 每轮追问都必须明显缩小研究范围，避免机械反问

### 交互限制

- 澄清阶段不展示执行进度条
- 不出现“研究中”状态文案
- 不显示运行期 artifact 面板

## 7.3 阶段三：计划确认

### 触发条件

- 系统已认为问题足够清晰，可形成可执行计划

### 展现方式

- 计划作为一条助手消息直接出现在对话流内
- 消息中包含：
  - 研究目标摘要
  - 计划概览
  - 子任务 / 研究维度
  - 主要来源路线
  - 必要时的预算或边界提示
- 计划消息下方提供唯一主 CTA：`确认并开始研究`

### 设计目标

- 计划必须“像消息”，不是游离在页面角落的工具卡片
- 用户在阅读对话上下文后，自然过渡到确认动作
- 视觉上强调这是“开始执行前的最后判断点”

### 不应出现的内容

- 允许用户再次切换“是否需要确认”
- 多个冲突的主按钮
- 计划在一边、确认按钮在另一边的割裂布局

## 7.4 阶段四：执行工作台

### 触发条件

- 用户已点击确认并开始研究

### 展现方式

- 页面从对话式规划界面切换到执行工作台
- 工作台继续复用现有三类能力区：
  - 研究轨道 / 状态摘要
  - 当前发现 / 报告阅读区
  - 事件与产物面板

### 体验目标

- 让用户清楚感知“现在已经不是规划，而是执行”
- 相比确认前阶段，提高信息密度，但保留视觉克制
- 工作台的重点是研究进展与结果阅读，而不是配置项堆积

## 8. 视觉设计系统

## 8.1 整体风格

- 暗色
- 安静
- 编辑感
- 偏研究出版物气质，而不是炫技式 AI 面板

## 8.2 色彩策略

- 背景以深蓝灰和近黑色层次为主
- 强调色仅用于主 CTA、关键状态和轻微焦点引导
- 避免常见紫粉 AI 渐变作为主叙事

## 8.3 字体策略

- 标题使用更具编辑感 / 学术感的 display serif 气质
- 正文与交互文案使用高可读的人文 sans
- 不采用过于通用且缺少性格的默认组合来主导首屏气质

## 8.4 组件风格

- 大圆角
- 轻边框
- 轻层次
- 少量必要阴影
- 动效以平滑过渡为主，避免炫技

## 8.5 视觉反模式

- 不做重玻璃态
- 不做强霓虹科技风
- 不做 dashboard 式多卡片空态
- 不做营销区块堆叠

## 9. 前端实现边界

## 9.1 ResearchPage

负责阶段状态机与页面切换：

- 无会话 / 空态
- 澄清中
- 等待确认
- 已确认执行

ResearchPage 不再暴露“是否执行前确认”的本地开关状态。

## 9.2 ResearchComposer

重做为极简研究入口组件，只承担：

- 标题
- 说明
- 问题输入
- 示例问题
- 发起规划动作

不再承担策略配置或运行控制配置。

## 9.3 澄清线程组件

新增或重构专门的对话线程组件，用于承载：

- 用户问题
- 助手澄清问题
- 用户补充回答
- 计划消息

该组件是确认前阶段的主容器。

## 9.4 PlanPreviewPanel

从“侧边计划卡片”调整为“助手消息内计划块”。

保留计划内容结构，但视觉宿主从 sidebar 转入对话流。

## 9.5 执行态组件

以下组件在确认后继续使用，但只服务执行态：

- ResearchWorkspaceShell
- ResearchSessionRail
- ResearchCanvas
- ArtifactPanel
- 进度 / 高级事件相关组件

## 10. 后端契约与状态要求

## 10.1 创建研究会话

前端创建会话时，不再提供“用户可切换的 require_confirmation 配置”。

产品语义固定为：

- 先规划
- 规划完成后等待确认

因此后端需要把“确认执行”视为默认强制流程，而不是复杂度触发的可选流程。

## 10.2 澄清能力

后端需要能表达两种 planner 输出：

1. **信息不足，需要继续澄清**
2. **信息充分，可生成计划并等待确认**

无论最终采用单接口还是多事件方案，前端都必须能区分这两类状态。

## 10.3 会话状态建议

当前状态语义需要收敛为更适合产品表达的主线：

- `planning`：正在分析问题
- `clarifying`：需要用户补充信息
- `awaiting_confirmation`：计划已生成，等待确认
- `queued`：用户已确认，准备执行
- `running` / `finalizing` / `final`：执行期状态

如果短期内不新增 `clarifying` 枚举，也必须通过现有事件明确表达“当前仍在追问，不可执行”。

## 10.4 confirm-plan 接口

该接口保留，并成为唯一进入执行阶段的正式入口。

不应再存在“前端直接创建后自动运行”的常规路径。

## 11. 错误处理

### 11.1 澄清阶段

- 若澄清生成失败，在对话流中展示可理解错误
- 用户可以继续补充或重试
- 不允许静默跳过澄清直接出计划

### 11.2 计划阶段

- 若计划生成失败，停留在确认前界面
- 允许重试计划生成
- 不允许用空计划或弱计划继续执行

### 11.3 执行阶段

- 仅在确认后才可能进入执行失败场景
- 执行失败展示在工作台中，不反向污染确认前体验

## 12. 验收标准

### 12.1 视觉验收

- 空态显著简化，首屏不出现旧配置开关
- 整体风格符合暗色、安静、编辑感
- 计划确认阶段视觉上属于对话流，而不是侧边孤立卡片

### 12.2 交互验收

- 输入一个明显模糊的问题时，系统先追问，不直接执行
- 输入一个信息充分的问题时，系统直接生成计划并等待确认
- 用户未确认前，研究不会进入 queued / running
- 用户确认后，页面切换到执行工作台

### 12.3 契约验收

- 前端不再提供“执行前确认”的可选配置
- 后端固定支持“澄清 / 计划确认 / 确认后执行”主链路
- confirm-plan 仍可用，并成为必经路径

## 13. 风险与应对

### 风险 1：当前后端 planner 只有“直接出计划”能力，缺少真正的澄清状态

应对：implementation plan 中优先补齐 planner 输出语义和前端状态映射，再做完整 UI 接入。

### 风险 2：现有 workbench 组件已在进行中，贸然全部推倒会扩大改动范围

应对：确认前阶段新增对话式壳层；执行态尽量复用现有 workbench，只调整切入时机和视觉承接。

### 风险 3：前端先改视觉但后端状态未收敛，会出现假交互

应对：以契约和状态机为先，禁止仅做静态界面伪装完成。

## 14. 推荐实施顺序

1. 收敛后端 planner / session 状态语义
2. 收敛前端 ResearchPage 状态机
3. 重做确认前阶段 UI（空态 + 澄清对话 + 计划消息）
4. 接入 confirm-plan 后的阶段切换
5. 调整执行态视觉承接与回归测试

## 15. 最终结论

本次深度研究首页重设计不只是“换皮”，而是把产品主路径从“立即执行研究”改为“先共同定义研究任务，再确认执行”。前半段采用对话式研究规划，后半段采用执行工作台，两者通过计划确认点完成切换。这样既保留了类似 ChatGPT / Gemini / Qwen 深度研究产品的自然交互，也保留了本项目执行阶段对结构化进度与产物阅读的要求。
