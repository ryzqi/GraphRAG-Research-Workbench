## 迁移项清单

### Milvus / pymilvus（2.6.7）

- [ ] **P0 `WeightedRanker` 传参方式与 pymilvus==2.6.7 不兼容（会直接 TypeError）**

  - 现状：`backend/src/app/integrations/milvus_client.py:443` 使用 `WeightedRanker([dense_weight, sparse_weight])`
  - 不符合点：pymilvus 2.6.7 的 `WeightedRanker` 构造函数是 `WeightedRanker(*nums, norm_score=True)`；传入 `list` 会触发类型错误（本地已在 `backend/.venv/Lib/site-packages/pymilvus/client/abstract.py` 验证）。
  - 推荐改法（目标）：按顺序传入权重参数，例如：
    ```python
    # reqs=[dense_req, sparse_req] -> 权重也按相同顺序传
    ranker_impl = WeightedRanker(dense_weight, sparse_weight)
    ```
  - 影响面：`backend/src/app/integrations/milvus_client.py:404` 的 `hybrid_search(...)` 路径（检索会在运行期崩溃）
  - Ref：`https://github.com/milvus-io/milvus-docs/blob/v2.6.x/site/en/tutorials/hybrid_search_with_milvus.md?plain=1#L182#run-the-search`
- [ ] **P1 AsyncMilvusClient 生命周期未关闭 + 进程级缓存存在跨事件循环复用风险**

  - 现状：
    - FastAPI 启动时创建 milvus client：`backend/src/app/main.py:35`（`app.state.milvus_client = MilvusClient()`），shutdown 未 close
    - 进程级缓存：`backend/src/app/integrations/milvus_client.py:634`（`@lru_cache`）+ `backend/src/app/integrations/milvus_client.py:635`（`def get_milvus_client()`）
    - Celery 任务使用 `asyncio.run(...)`（每任务创建新 event loop）：例如 `backend/src/app/worker/tasks/research.py:35`，但依赖同一个缓存 milvus client：`backend/src/app/worker/tasks/research.py:76`
  - 风险/不符合点：
    - 资源释放缺失（连接长期占用、进程退出前不释放）
    - cached async client 在不同 event loop 中复用，存在运行期报错/不稳定风险（与 redis.asyncio 同类问题）
  - 推荐改法（目标）：
    - FastAPI：在 `lifespan` 的 `yield` 后显式 close，并清理缓存
    - Celery：不要复用进程级 cached milvus async client；改为“每任务创建 + finally close”
  - 影响面（部分调用点）：
    - `backend/src/app/api/v1/endpoints/health.py:54`
    - `backend/src/app/worker/tasks/ingestion.py:64`
    - `backend/src/app/worker/tasks/index_rebuild.py:60`
    - `backend/src/app/worker/tasks/research.py:76`
    - `backend/src/app/worker/tasks/evaluation.py:48`
  - Ref：`https://github.com/milvus-io/milvus-docs/blob/v2.6.x/site/en/tutorials/use-async-milvus-client-with-asyncio.md?plain=1#L91#connect-with-asyncmilvusclient`

### Redis（redis-py 7.1.0）/ Celery（5.6.2）

- [ ] **P0 `redis.asyncio.Redis` 被 `@lru_cache` 缓存并跨 `asyncio.run` 事件循环复用（高概率运行期故障）**

  - 现状：
    - Redis 单例：`backend/src/app/integrations/redis_client.py:13`（`@lru_cache`）+ `backend/src/app/integrations/redis_client.py:14`（`def get_redis()`）
    - Celery 任务入口是同步 task + `asyncio.run(...)`：例如 `backend/src/app/worker/tasks/evaluation.py:25` / `backend/src/app/worker/tasks/research.py:35`
    - Celery 任务内部使用 cached redis client：`backend/src/app/worker/tasks/evaluation.py:50`、`backend/src/app/worker/tasks/research.py:79`
  - 风险/不符合点：
    - `redis.asyncio` 客户端与事件循环强绑定；跨 loop 复用常见报错包括 “event loop is closed / attached to a different loop”
    - 未显式 close/释放连接池，存在连接泄漏风险
  - 推荐改法（目标）：
    - 将 Redis 客户端拆为两类：
      1) FastAPI（单 event loop）可持久化/缓存，并在 shutdown `await redis.aclose()` + cache_clear
      2) Celery（`asyncio.run` 每次新 loop）必须“每任务创建 + finally close”
  - 影响面（部分调用点）：
    - `backend/src/app/api/v1/endpoints/health.py:50`
    - `backend/src/app/services/kb_chat_service.py:90`
    - `backend/src/app/agents/tools/web_search.py:446`
    - `backend/src/app/worker/tasks/evaluation.py:50`
    - `backend/src/app/worker/tasks/research.py:79`
  - Ref：`https://redis.io/docs/latest/develop/clients/redis-py/connect#connect-with-a-connection-pool`
- [ ] **P2 Celery 5.6 配置建议：显式设置 startup 重试与序列化策略（降低行为漂移/安全风险）**

  - 现状：`backend/src/app/worker/celery_app.py:22` 仅设置 `task_track_started/timezone/time_limit`
  - 建议（目标）：
    - 显式设置：`broker_connection_retry_on_startup=True`
    - 显式约束序列化：`accept_content=["json"]`、`task_serializer="json"`、`result_serializer="json"`、`result_accept_content=["json"]`
    - 若业务不读取任务结果：建议设置 `task_ignore_result=True`；否则至少设置 `result_expires=<seconds>`
  - Ref：
    - `https://docs.celeryq.dev/en/stable/userguide/configuration.html#broker-connection-retry-on-startup`
    - `https://docs.celeryq.dev/en/stable/userguide/configuration.html#broker-connection-retry`

### SQLAlchemy（2.0.46）/ Alembic（1.18.1）

- [ ] **P2 AsyncEngine 生命周期建议：应用 shutdown 时 dispose + 清理缓存**
  - 现状：AsyncEngine 通过 `@lru_cache` 常驻：`backend/src/app/db/session.py:17`（`def get_engine()`），未在 FastAPI shutdown 时 `await engine.dispose()`，也未 `cache_clear()`
  - 建议（目标）：在 `backend/src/app/main.py` 的 `lifespan` shutdown（`yield` 后）集中释放底层资源（engine / redis / http client / milvus 等）
  - Ref：`https://docs.sqlalchemy.org/en/20/orm/extensions/asyncio.html#asynchronous-i-o-asyncio`

### httpx（0.28.1）

- [ ] **P2 共享 HTTP 客户端存在未关闭路径（资源释放一致性）**
  - 现状：
    - `backend/src/app/integrations/http_client.py:33` 提供 `get_shared_http_client()`（`@lru_cache`）
    - `backend/src/app/agents/tools/web_search.py:545` 使用 `get_shared_http_client()`
    - 虽然提供了关闭函数：`backend/src/app/integrations/http_client.py:38`（`close_shared_http_client()`），但 FastAPI lifespan 未调用
  - 建议（目标）：
    - 统一通过 `app.state.http_client` 注入，避免“多套 client 并存”
    - 或在 shutdown 中调用 `close_shared_http_client()`，并清理缓存
  - Ref：`https://github.com/encode/httpx/blob/master/docs/async.md?plain=1#L1#async-support`
